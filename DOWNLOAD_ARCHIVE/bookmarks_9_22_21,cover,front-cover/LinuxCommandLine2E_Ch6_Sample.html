Standard Input, Output, and Error
Many of the programs that we have used so far produce output of some
kind. This output often consists of two types.
The programs results; that is, the data the program is designed to
produce
Status and error messages that tell us how the program is getting along
If we look at a command like
ls, we can see that it displays its results
and its error messages on the screen.
Keeping with the Unix theme of everything is a le, programs such
as ls actually send their results to a special le called standard output (often
expressed as stdout) and their status messages to another le called standard
error (stderr). By default, both standard output and standard error are linked
to the screen and not saved into a disk le.
In addition, many programs take input from a facility called standard
input (stdin), which is, by default, attached to the keyboard.
I/O redirection allows us to change where output goes and where input
comes from. Normally, output goes to the screen and input comes from the
keyboard, but with I/O redirection, we can change that.
Redirecting Standard Output
I/O redirection allows us to redene where standard output goes. To
redirect standard output to another le instead of the screen, we use the >
redirection operator followed by the name of the le. Why would we want
to do this? Its often useful to store the output of a command in a le. For
example, we could tell the shell to send the output of the ls command to
the le ls-output.txt instead of the screen.
[me@linuxbox ~]$ ls -l /usr/bin > ls-output.txt
Here, we created a long listing of the /usr/bin directory and sent the
results to the le ls-output.txt. Lets examine the redirected output of the
command, shown here:
[me@linuxbox ~]$ ls -l ls-output.txt
-rw-rw-r-- 1 me me 167878 2018-02-01 15:07 ls-output.txt
Gooda nice, large, text le. If we look at the le with less, we will
see that the le ls-output.txt does indeed contain the results from our ls
command.
[me@linuxbox ~]$ less ls-output.txt
Now, lets repeat our redirection test, but this time with a twist. Well
change the name of the directory to one that does not exist.

[me@linuxbox ~]$ ls -l /bin/usr > ls-output.txt

We received an error message. This makes sense since we specied the
nonexistent directory /bin/usr, but why was the error message displayed on
the screen rather than being redirected to the le ls-output.txt? The answer
is that the ls program does not send its error messages to standard output.
Instead, like most well-written Unix programs, it sends its error messages
to standard error. Because we redirected only standard output and not
standard error, the error message was still sent to the screen. Well see how
to redirect standard error in just a minute, but rst lets look at what hap-
pened to our output le.
[me@linuxbox ~]$ ls -l ls-output.txt
-rw-rw-r-- 1 me me 0 2018-02-01 15:08 ls-output.txt
The le now has zero length! This is because when we redirect output
with the > redirection operator, the destination le is always rewritten from
the beginning. Because our ls command generated no results and only an
error message, the redirection operation started to rewrite the le and then
stopped because of the error, resulting in its truncation. In fact, if we ever
need to actually truncate a le (or create a new, empty le), we can use a
trick like this:
[me@linuxbox ~]$ > ls-output.txt
Simply using the redirection operator with no command preceding it
will truncate an existing le or create a new, empty le.
So, how can we append redirected output to a le instead of overwriting
the le from the beginning? For that, we use the >> redirection operator,
like so:
[me@linuxbox ~]$ ls -l /usr/bin >> ls-output.txt
Using the >> operator will result in the output being appended to the le.
If the le does not already exist, it is created just as though the > operator had
been used. Lets put it to the test.
[me@linuxbox ~]$ ls -l /usr/bin >> ls-output.txt
[me@linuxbox ~]$ ls -l /usr/bin >> ls-output.txt
[me@linuxbox ~]$ ls -l /usr/bin >> ls-output.txt
[me@linuxbox ~]$ ls -l ls-output.txt
-rw-rw-r-- 1 me me 503634 2018-02-01 15:45 ls-output.txt
We repeated the command three times, resulting in an output le
three times as large.

Redirecting Standard Error
Redirecting standard error lacks the ease of a dedicated redirection opera-
tor. To redirect standard error, we must refer to its le descriptor. A program
can produce output on any of several numbered le streams. While we have
referred to the rst three of these le streams as standard input, output,
and error, the shell references them internally as le descriptors 0, 1, and
2, respectively. The shell provides a notation for redirecting les using the
le descriptor number. Because standard error is the same as le descriptor
number 2, we can redirect standard error with this notation:
[me@linuxbox ~]$ ls -l /bin/usr 2> ls-error.txt
The le descriptor 2 is placed immediately before the redirection oper-
ator to perform the redirection of standard error to the le ls-error.txt.
Redirecting Standard Output and Standard Error to One File
There are cases in which we may want to capture all of the output of a com-
mand to a single le. To do this, we must redirect both standard output and
standard error at the same time. There are two ways to do this. Shown here
is the traditional way, which works with old versions of the shell:
[me@linuxbox ~]$ ls -l /bin/usr > ls-output.txt 2>&1
Using this method, we perform two redirections. First we redirect
standard output to the le ls-output.txt, and then we redirect le descrip-
tor 2 (standard error) to le descriptor 1 (standard output) using the
notation 2>&1.

Disposing of Unwanted Output
Sometimes silence is golden and we dont want output from a command;
we just want to throw it away. This applies particularly to error and status
messages. The system provides a way to do this by redirecting output to a
special le called /dev/null. This le is a system device often referred to as
a bit bucket, which accepts input and does nothing with it. To suppress error
messages from a command, we do this:
[me@linuxbox ~]$ ls -l /bin/usr 2> /dev/null

Pipelines
The capability of commands to read data from standard input and send to
standard output is utilized by a shell feature called pipelines. Using the pipe
operator |, the standard output of one command can be piped into the stan-
dard input of another.
command1 | command2
To fully demonstrate this, we are going to need some commands.
Remember how we said there was one we already knew that accepts stan-
dard input? Its less. We can use less to display, page by page, the output
of any command that sends its results to standard output.
[me@linuxbox ~]$ ls -l /usr/bin | less
This is extremely handy! Using this technique, we can conveniently
examine the output of any command that produces standard output.
